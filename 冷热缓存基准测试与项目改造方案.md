# Block3D 冷热缓存基准测试与项目改造方案

> 文档状态：设计草案  
> 编写日期：2026-07-16  
> 适用项目：`block3d-cpp`  
> 目的：作为后续 benchmark 实验环境改造、冷路径性能优化和赛事提交优化的统一设计依据。

---

## 1. 背景

赛事需求明确说明：正式测试前会清理缓存，以保证测试环境一致和公平。因此，正式评分更接近冷缓存或首次访问场景，而当前项目主要通过 `--warm-up` 观察热缓存性能，容易高估正式环境中的读取速度。

项目需要同时建立两类性能基线：

1. **冷缓存性能**：本地以大数据包冲刷成功后的第一次访问作为冷缓存口径。
2. **热缓存性能**：反映数据已进入 OS 页缓存后，算法本身的 CPU、内存访问和切片提取上限。

本地 Windows 日常开发统一采用稳定、可重复的大文件缓存冲刷：只要无关大数据包按要求完整读取并冲刷成功，该轮 benchmark 就按本项目的冷缓存环境处理，不需要额外的系统级缓存验证。

## 2. 当前项目行为确认

### 2.1 `BlockedFileReader` 构造时不会自动预热

当前 `BlockedFileReader` 构造过程只执行：

- 读取 `.b3d` 文件头；
- 读取块偏移索引表；
- mmap 整个文件；
- 初始化 reader 内部状态。

它不会自动遍历或预取 `.b3d` 数据区。完整数据预热只有显式调用以下接口时才发生：

```cpp
reader.warm_up(...);
```

因此，后续不应把完整预热加入 reader 构造函数。否则会：

- 破坏冷缓存测试能力；
- 让每次打开 18 GB/50 GB 文件都产生大量隐藏 I/O；
- 混淆 reader 初始化时间、预热时间和正式读取时间；
- 使正式提交程序可能在计时前执行不被允许的隐藏读取。

### 2.2 当前两个 benchmark 的计时口径不同

- `block3d_cli bench`：只统计切片读取，不写出结果文件。
- `run_test`：统计切片读取和结果文件写出，并可包含 `_commit`/`fsync`。

后续改造必须保留这个区别，不能把两者的结果直接混合比较。

### 2.3 当前 warm-up 的限制

Windows 的 `PrefetchVirtualMemory` 和 Linux 的 `madvise(MADV_WILLNEED)` 都属于预取提示。接口返回成功不等于所有目标页已经真实进入物理内存。

后续应把 `warm_up()` 的完成语义增强为：

```text
提交系统预取提示
  -> 实际顺序触碰目标范围中的每个页面
  -> 触碰范围最后一个字节
  -> 完成后才设置 warm_up_done
```

## 3. 术语与缓存状态定义

后续文档和程序输出统一使用以下术语。为便于开发和复现实验，只要大数据包冲刷步骤成功完成，就把该轮结果作为本项目的冷缓存结果使用。

### 3.1 `cold_scrubbed`

执行 benchmark 前，使用无关大数据包逐页读取，对 OS 文件页缓存制造足够的替换压力，然后重新创建 reader 并执行请求。

这是本地日常开发的默认冷缓存环境。只要 scrub 文件容量满足要求、读取覆盖完整、checksum 输出正常、过程无错误，该状态即视为冷缓存。

### 3.2 `cold_first_touch`

没有主动预热目标文件，但也没有执行大数据包冲刷。

该状态只作为兼容或临时测试使用，不作为本项目主要冷缓存口径。

### 3.3 `hot_prefetched`

显式预热完成后执行相同请求计划。

热状态继续区分：

- `full_dataset`：尝试预热整个 `.b3d` 数据区；
- `workload_set`：只预热当前请求计划会访问的物理块集合；
- `partial`：受 `--warmup-memory` 或物理内存限制，只预热部分数据；
- `best_effort`：已完整扫描，但无法保证全部目标页仍驻留在 RAM 中。

在本地 32 GB 内存上，50 GB 数据集不可能完整常驻，因此热缓存结果主要用于观察算法上限和重复访问场景。

## 4. 总体 benchmark 流程

同一数据包的冷、热阶段必须串行执行，不能并发执行。并发执行会使热阶段立即污染冷阶段。

推荐的单轮流程：

```text
[阶段 0] 准备
  - 校验 .b3d 和 scrub 文件存在
  - 生成一次固定请求计划
  - 打印 plan_hash

[阶段 1] 冷缓存冲刷
  - 销毁旧 reader 和大型结果缓冲
  - 逐页读取无关 scrub 文件
  - 打印 scrub bytes / elapsed / checksum
  - 关闭 scrub 文件和映射
  - 等待存储队列短暂稳定

[阶段 2] 冷缓存 benchmark
  - 创建新 reader
  - 不调用 warm_up()
  - 使用固定请求计划执行 cold_scrubbed 阶段
  - 输出 cold_scrubbed 结果
  - 销毁 cold reader

[阶段 3] 热缓存准备
  - 创建临时 reader
  - 显式预热目标数据或工作集
  - 打印 warm-up bytes / elapsed
  - 销毁临时 reader

[阶段 4] 热缓存 benchmark
  - 创建新 reader
  - 使用同一请求计划执行 hot_prefetched 阶段
  - 输出 hot_prefetched 结果

[阶段 5] 汇总
  - 输出 cold/hot 对比
  - 输出输出文件位置和结果校验信息
```

多轮实验时重复完整阶段：

```text
准备 -> scrub -> cold -> warm-up -> hot -> 汇总
准备 -> scrub -> cold -> warm-up -> hot -> 汇总
准备 -> scrub -> cold -> warm-up -> hot -> 汇总
```

建议至少运行 3 轮。日常判断可看单轮结果，关键优化对比时使用多轮中位数。

## 5. 固定请求计划

冷、热阶段必须使用完全相同的工作负载，包括：

- 随机种子；
- 切片坐标；
- 请求顺序；
- 重复坐标；
- axis 顺序；
- random/sequential 模式；
- 线程数；
- 输出和同步策略。

建议增加内部结构：

```cpp
struct BenchCasePlan {
    Axis axis;
    BenchMode mode;
    std::vector<std::uint64_t> indices;
    std::uint64_t plan_hash;
};
```

请求计划只生成一次。冷、热阶段都读取同一个不可变计划，不允许各自重新运行随机数生成器。

输出 `plan_hash`，用于确认两阶段实际执行的是同一组请求。

## 6. 本地模拟冷缓存：大文件缓存冲刷

### 6.1 原理

Windows 中 `.b3d` 的 mmap 页面和普通缓存文件页面都会占用物理内存。读取一个大于可用于文件缓存内存容量的无关文件，并实际触碰其中的所有页面，会迫使 Windows 回收一部分旧文件页，其中通常包括之前访问过的 `.b3d` 页面。

该方法不能保证清除：

- SSD/HDD 控制器缓存；
- RAID 缓存；
- 集群存储服务端缓存；
- 网络存储缓存；
- 全部文件系统元数据缓存。

在本项目的本地实验中，只要无关大数据包冲刷成功，就把后续第一次访问作为冷缓存结果使用。

### 6.2 scrub 文件大小

scrub 文件大小主要由物理内存决定，而不是由 `.b3d` 文件大小决定。

推荐配置：

| 强度 | scrub 文件大小 |
|---|---:|
| 快速 | `1.25 × 物理内存` |
| 默认 | `1.50 × 物理内存` |
| 强力 | `2.00 × 物理内存` |

本地 32 GB 内存机器建议：

- 快速：40 GB；
- 默认：48 GB；
- 强力：64 GB。

默认采用 `1.50 × 物理内存`，本地 32 GB 机器即 48 GB。若磁盘空间不足，可临时使用 `1.25 × 物理内存`，但程序输出应记录实际 scrub 字节数。

### 6.3 不要反复读取一个小文件

例如，在 32 GB 内存机器上反复读取 8 GB 文件 10 次，不等价于读取 80 GB 的唯一页面。第一次之后，这 8 GB 文件本身可能已经被缓存，后续读取只是在重复访问相同页面，无法持续产生足够的替换压力。

优先策略是扩大唯一页面总量，而不是增加小文件重复读取次数。

默认 `passes=1`。多 pass 只作为实验参数，不作为弥补文件容量不足的主要方式。

### 6.4 scrub 文件创建要求

scrub 文件只创建一次，后续重复只读使用。不要在每轮 benchmark 前重新生成，否则会引入：

- 大量脏页；
- 后台写回；
- 额外 SSD/HDD 写入；
- 文件系统元数据更新；
- 对正式读取阶段的 I/O 干扰。

scrub 文件应满足：

- 真实分配磁盘空间；
- 不是稀疏文件；
- 不启用 NTFS 压缩；
- 不位于 RAM disk；
- 内容不应是容易被压缩或去重的全零数据；
- 使用确定性伪随机块分批写入，避免生成阶段占用大量内存。

如果单个磁盘空间不足，也可以读取多个无关文件，只要唯一页面总量达到目标容量。

### 6.5 scrub 文件读取方式

推荐使用普通只读 mmap 或普通 buffered I/O，并真实触碰每个系统页面。

不要使用：

- Windows `FILE_FLAG_NO_BUFFERING`；
- Linux `O_DIRECT`；
- 只调用 `PrefetchVirtualMemory` 而不触页；
- 只调用 `madvise` 而不触页。

无缓存 I/O 会绕过或弱化 OS 文件页缓存，不适合用来制造缓存替换压力。

伪代码：

```cpp
std::uint64_t checksum = 0;
const std::size_t page_size = query_system_page_size();

for (std::uint64_t offset = 0; offset < file_size; offset += page_size) {
    checksum += static_cast<std::uint8_t>(mapped_data[offset]);
}

if (file_size != 0) {
    checksum += static_cast<std::uint8_t>(mapped_data[file_size - 1]);
}

print_checksum(checksum);
```

要求：

- 页面大小从操作系统查询；
- 访问最后一个字节，避免尾页遗漏；
- checksum 必须被实际使用，防止优化器删除读取；
- 大文件可以分段映射，例如每段 1～4 GB；
- 结束后关闭 view、mapping 和文件句柄；
- scrub 前销毁旧 reader 和大型切片结果缓冲。

### 6.6 scrub 文件放置位置

#### 与 `.b3d` 位于同一物理设备

优点：

- 能冲刷 OS 页缓存；
- 更可能扰动目标设备自身的小容量缓存；
- 更接近完整存储链路的冷访问。

缺点：

- 可能改变 HDD 磁头位置；
- 可能引起 SSD 升温；
- scrub 完成时设备队列可能仍有请求。

建议 scrub 后等待 500～2000 ms，默认可设为 1000 ms。等待不计入 benchmark。

#### 位于另一物理设备

仍然可以制造全局物理内存压力，驱逐目标文件的 OS 缓存，同时减少目标设备温度和队列扰动。

这种方式主要模拟 OS cache cold，不能期望冲刷目标设备控制器缓存。

本地默认优先使用同一物理设备；如果发现 SSD 热降速明显，再切换到另一设备并在结果中记录。

## 7. 冷缓存隔离粒度

### 7.1 `suite` 模式

```text
scrub 一次
  -> random X/Y/Z
  -> sequential X/Y/Z
```

这更接近“正式测试前统一清一次缓存，然后运行整套工作负载”的场景，也是默认模式。

需要注意：只有 suite 的第一个 case 从完整模拟冷状态开始，后续 case 会受到前面请求的缓存影响。

### 7.2 `case` 模式

```text
scrub -> random X
scrub -> random Y
scrub -> random Z
...
```

每个 axis/mode 都尽可能独立冷启动，适合定位单项瓶颈，但测试时间很长。

建议默认：

```text
--cold-isolation suite
```

性能分析或块大小调优时使用：

```text
--cold-isolation case
```

## 8. 建议的命令行契约

### 8.1 缓存模式

```text
--cache-mode both|cold|hot
```

| 模式 | 行为 |
|---|---|
| `both` | 先执行冷阶段，再预热并执行热阶段 |
| `cold` | 只执行冷阶段 |
| `hot` | 先预热，再执行热阶段 |

目标状态是两个 benchmark 入口默认执行 `both`。为了兼容旧脚本：

- 单独使用 `--warm-up` 时，保持“只执行热阶段”的旧语义；
- `--warm-up` 可视为 `--cache-mode hot` 的兼容别名；
- `--warm-up` 与 `--cache-mode cold/both` 同时出现时应报参数冲突；
- 迁移期输出兼容提示，后续再决定是否弃用 `--warm-up`。

### 8.2 冷缓存参数

建议支持：

```text
--cold-method scrub|none
--cold-scrub-file <path>
--cold-scrub-ratio <float>
--cold-scrub-passes <N>
--cold-settle-ms <N>
--cold-isolation suite|case
```

语义：

- `scrub`：程序读取指定无关大数据包，生成 `cold_scrubbed`，这是默认冷缓存口径；
- `none`：不清缓存且不预热，生成 `cold_first_touch`，只用于兼容或临时测试。

建议默认：

```text
cold method:       scrub
scrub ratio:       1.50
scrub passes:      1
settle time:       1000 ms
cold isolation:    suite
```

如果 `--cache-mode` 包含冷阶段且 `--cold-method scrub`，则必须提供有效 scrub 文件。文件不存在、容量不足、读取失败或 checksum 过程异常时，本轮 cold benchmark 应直接失败，不应静默降级为 `cold_first_touch`。

### 8.3 scrub 文件准备

建议增加独立命令，而不是在 benchmark 开始前自动创建：

```text
block3d_cli cache-prepare <file> --size auto
block3d_cli cache-prepare <file> --size-gb 48
```

`--size auto` 根据物理内存和 `--cold-scrub-ratio` 计算目标大小。

创建完成后程序退出，正式 benchmark 在下一次进程运行中使用该文件。

### 8.4 热缓存范围

建议增加：

```text
--warmup-scope dataset|workload
```

- `dataset`：预热整个 `.b3d` 数据区，对应当前 `warm_up()` 方向；
- `workload`：预热固定请求计划会访问的物理块并集。

本地内存不足以容纳整个数据集时，推荐使用 `workload`，它更接近“相同请求重复执行后的热状态”。输出必须明确这不是完整数据集热缓存。

## 9. 两个 benchmark 入口的改造

### 9.1 `block3d_cli bench`

建议拆分为：

```text
build_bench_plan()
run_cache_scrub()
warm_up_for_plan()
run_read_phase()
print_phase_results()
print_phase_comparison()
```

计时范围继续保持只读：

- reader 构造不计时；
- 请求计划生成不计时；
- cache scrub 不计时，但单独报告耗时；
- warm-up 不计时，但单独报告耗时；
- 只统计 `read_slices_batch()` 或后续融合批量读取实现的时间。

冷、热阶段分别创建新的 reader，避免把 reader 内部块列表缓存收益混入 OS 页缓存对比。

### 9.2 `run_test`

`run_test` 的主指标必须继续符合赛事“完成写出为结束标志”的要求。

建议扩展结果：

```cpp
struct BenchResult {
    double read_sec;
    double write_sec;
    double total_sec;
    // existing counts and throughput fields
};
```

正式比较仍使用 `total_sec`，拆分时间只用于定位瓶颈。

`both` 模式使用独立输出目录：

```text
<run_dir>/cold_scrubbed/
<run_dir>/hot_prefetched/
```

不能让 hot 阶段覆盖 cold 阶段文件，否则 hot 阶段会受益于：

- 已存在的文件；
- 已分配的磁盘空间；
- 已缓存的目录元数据；
- 已存在的输出文件页。

两个目录应在计时前同时创建。正式运行前根据请求数量和切片大小估算输出总量，并使用 `std::filesystem::space()` 检查磁盘空间。

## 10. 结果输出格式

保留当前人类可读表格，同时增加稳定的机器可读行。

示例：

```text
CACHE_SCRUB method=unrelated_file_sweep
CACHE_SCRUB file=D:\cache\block3d-scrub.bin
CACHE_SCRUB bytes=51539607552 ram_ratio=1.50 passes=1
CACHE_SCRUB elapsed_sec=18.421 checksum=123456

CACHE_PHASE name=cold_scrubbed
CACHE_VALIDITY state=scrub_success
TIMING_SCOPE value=read_only
PLAN_HASH value=8f4c30a1

BENCHMARK_RESULT cache=cold_scrubbed mode=random axis=x total_sec=...
BENCHMARK_RESULT cache=cold_scrubbed mode=random axis=y total_sec=...
BENCHMARK_RESULT cache=cold_scrubbed mode=random axis=z total_sec=...

WARMUP_RESULT scope=workload elapsed_sec=... bytes=...

CACHE_PHASE name=hot_prefetched
BENCHMARK_RESULT cache=hot_prefetched mode=random axis=x total_sec=...
BENCHMARK_RESULT cache=hot_prefetched mode=random axis=y total_sec=...
BENCHMARK_RESULT cache=hot_prefetched mode=random axis=z total_sec=...

BENCHMARK_COMPARE mode=random axis=x observed_ratio=3.27
```

统一定义：

```text
observed_ratio = cold_time / hot_time
```

使用 `observed_ratio`，用于表示本项目固定 scrub 冷缓存口径与热缓存口径之间的时间比值。

结果至少记录：

- cache state；
- cache validity；
- scrub 文件和读取字节数；
- scrub/RAM 比值；
- warm-up scope；
- timing scope；
- plan hash；
- 线程数；
- 请求数量；
- 总读取或写出字节数；
- 是否执行同步写盘；
- 数据集和 block size；
- storage class；
- cold isolation 模式。

## 11. 建议的代码组织

缓存实验逻辑不属于 `.b3d` 公共格式，也不应成为 `BlockedFileReader` 构造行为。

建议增加 benchmark 私有支持模块：

```text
src/benchmark_cache.hpp
src/benchmark_cache.cpp
```

职责：

- 缓存模式和参数解析；
- 查询物理内存、页面大小；
- scrub 文件校验与创建；
- 分段 mmap/逐页触碰；
- 请求计划哈希；
- phase 标签和机器可读输出；
- warm-up scope 描述。

reader 公共 API 只保留通用读取和显式预热能力。若实现 workload warm-up，可选择：

1. 增加明确的通用范围预取 API；或
2. 由 benchmark 支持模块根据请求计划计算物理块范围后调用内部预取辅助函数。

不得为了 benchmark 方便而把自动 scrub 或自动完整预热塞入 reader 构造函数。

## 12. 正确性和自动测试

### 12.1 reader/warm-up 测试

在库测试中增加：

- 同步 `warm_up(false)` 完成后状态正确；
- 异步 warm-up 配合 `wait_warm_up()` 正常完成；
- warm-up 前后读取结果完全一致；
- memory cap 不越过文件数据区；
- 页 stride 和尾页处理正确。

不要在 CTest 中断言 hot 必须比 cold 快。小数据集、CI 缓存状态、Windows Defender 和后台进程都会导致这种时间断言不稳定。

### 12.2 benchmark 进程测试

使用小数据集通过真实进程验证：

1. `both` 输出 cold/hot 两组结果；
2. 两组 `plan_hash` 相同；
3. `--warm-up` 兼容模式只输出 hot；
4. 非法 `--cache-mode` 返回非零退出码；
5. 冲突参数返回清晰错误；
6. scrub 文件不存在或过小时给出错误/弱模拟警告；
7. `run_test` 生成两个 phase 目录；
8. 冷、热对应输出文件字节完全一致；
9. 测试输出结构，不测试绝对性能。

### 12.3 scrub 工具测试

测试：

- 自动容量计算；
- 小型测试文件逐页访问和尾页访问；
- checksum 可重复；
- 文件不足时的错误或警告；
- 非稀疏/压缩要求的诊断信息；
- 分段映射边界；
- 文件路径包含空格和非 ASCII 字符。

自动测试只验证 scrub 操作被正确执行，不断言操作系统一定驱逐了某个指定页面。

## 13. 本地 Windows 推荐操作流程

### 13.1 首次准备

以普通权限创建一次 scrub 文件：

```powershell
$exe = Join-Path $PWD "build\Release\block3d_cli.exe"
$scrub = "D:\block3d-cache\scrub.bin"

& $exe cache-prepare $scrub --size auto --cold-scrub-ratio 1.5
if ($LASTEXITCODE -ne 0) {
    throw "Failed to prepare cache scrub file: $LASTEXITCODE"
}
```

实际命令名称以最终实现为准。

### 13.2 日常测试

```powershell
$exe = Join-Path $PWD "build\Release\block3d_cli.exe"
$b3d = "D:\data\test18.b3d"
$scrub = "D:\block3d-cache\scrub.bin"

& $exe bench $b3d `

if ($LASTEXITCODE -ne 0) {
    throw "Benchmark failed: $LASTEXITCODE"
}
```

### 13.3 日常使用规则

本项目不要求日常 benchmark 使用 RAMMap、重启或其它管理员级缓存清理。只要 scrub 文件按要求完整读取成功，该轮 cold 阶段就按 `cold_scrubbed` 处理。

benchmark 程序本身不应要求管理员权限，也不应把管理员级缓存操作作为赛事提交的一部分。

## 14. scrub 默认规则与成功判定

本项目不再要求对本地环境进行冷缓存校准。日常 benchmark 采用固定规则：

```text
scrub_size = 1.50 × 物理内存
scrub_passes = 1
cold_settle_ms = 1000
```

本地 32 GB 内存机器默认使用约 48 GB scrub 文件。若磁盘空间不足，可临时降到 `1.25 × 物理内存`，但输出必须记录实际 scrub 字节数。

一次 scrub 成功必须同时满足：

1. scrub 文件存在；
2. scrub 文件大小达到本轮配置要求；
3. 文件不是空文件，且读取范围覆盖整个文件；
4. 逐页触碰过程完成；
5. 尾页最后一个字节被访问；
6. checksum 被计算并输出；
7. 没有 mmap、ReadFile、pread、短读或其它 I/O 错误；
8. scrub 后按配置等待 `cold_settle_ms`。

只要以上条件满足，后续 cold 阶段就记为 `cold_scrubbed`。如果任一条件失败，本轮 cold benchmark 失败，不降级、不继续输出冷缓存成绩。

本地优化的主要判断标准是：

> 同一机器、同一 scrub 配置、同一请求计划下，新实现是否稳定优于旧实现。

不要用本地绝对秒数预测官方机器绝对成绩。

## 15. 赛事提交时接近热缓存速度的优化方向

本地 scrub 和 warm-up 都是实验手段，不应作为正式提交程序隐藏在计时前执行。官方清缓存后，要缩小 cold/hot 差距，只能优化正式冷路径。

核心目标：

> 对完整请求集合尽量只读取一次必要数据，把离散访问组织成少量大范围顺序 I/O，并让读取、提取和写出互相重叠。

### 15.1 第一优先级：真正融合批量切片

当前 `read_slices_batch()` 主要是顺序派发单切片读取，没有充分利用同一 batch 中的块复用。

建议：

1. 按 `(axis, index / block_size)` 对请求分组；
2. 同一块层的物理块列表只计算一次；
3. 每个物理块只访问一次；
4. 从一个块中同时提取所有目标局部平面；
5. 按原始请求顺序回填结果，包括重复 index。

这项优化能同时减少：

- 冷缓存磁盘读取；
- 硬页错误；
- 重复块访问；
- `memcpy` 和循环开销；
- 函数调用和线程调度。

### 15.2 第二优先级：块并集和动态读取策略

对一个 batch 先计算所需物理块并集。

- 覆盖率低：按物理偏移排序并合并相邻范围，只读必要数据；
- 覆盖率高：顺序扫描数据区一次，在扫描过程中提取所有目标切片。

HDD 和集群存储通常更偏好少量大范围顺序读取；SSD/NVMe 更能承受离散并发，但仍需减少无效传输。

如果官方一次提供 X/Y/Z 全部请求，可进一步计算跨轴全局块并集，尽量让整个工作负载只承担一次冷 I/O 成本。

### 15.3 第三优先级：有界读写流水线

避免一次性保存整个 `vector<vector<float>>` batch。使用 2～4 个槽位：

```text
读取/提取切片 N+1
        与
写出/同步切片 N
并行重叠
```

收益：

- 控制内存峰值；
- 避免结果缓冲挤压输入文件页缓存；
- 重叠输入读取和输出写入；
- 更符合“完成写出才结束”的评分口径。

### 15.4 第四优先级：输出路径优化

根据赛事对“完成写出”的准确要求评估：

- 大块对齐写缓冲；
- 预分配输出文件；
- 异步写出；
- 减少不必要格式转换和复制；
- 是否真的需要逐文件 `_commit`/`fsync`。

不能通过省略赛事要求的持久化语义制造虚假成绩。

普通缓存写会让输出页占用 RAM，可能驱逐输入页。后续可 A/B 测试可选的 direct/no-buffering 输出后端，但必须正确处理对齐和尾块，并保证输出正确性。

### 15.5 第五优先级：计时范围内定向预取

正式处理请求后：

1. 计算下一批物理范围；
2. 合并相邻范围；
3. 提交异步预取；
4. CPU 同时处理已经就绪的块。

平台方向：

- Windows：`PrefetchVirtualMemory` 或 overlapped `ReadFile`；
- Linux/CentOS 7：`posix_fadvise`、线程池 `pread`；
- HDD：低队列深度、大连续范围；
- SSD/NVMe：适当提高 outstanding I/O；
- 集群存储：减少小请求和 RPC 数量。

这些预取操作必须位于正式计时范围内，除非赛事规则明确允许初始化阶段预取。

### 15.6 第六优先级：固定线程池和缓冲复用

避免单切片处理中频繁创建和 join 线程：

- reader 或 benchmark 生命周期内维护线程池；
- 每线程复用 scratch buffer；
- CPU 线程数与 I/O 队列深度分别配置；
- HDD 降低随机读并发；
- NVMe 根据实测增加并发。

### 15.7 第七优先级：重新选择 block size

块大小不只根据单切片触及块数选择，还应考虑：

- 随机 100 切片的块并集；
- 连续 10 切片的块复用；
- HDD 寻道成本；
- SSD/NVMe 读取放大；
- 集群存储请求粒度。

后续 block size 调整以 `cold_scrubbed` benchmark 作为主要依据。不同介质可以保留不同推荐配置。

### 15.8 后续方向：按块自适应压缩

可评估每块选择：

```text
raw
LZ4
bitshuffle + LZ4
```

只在压缩收益足够时保存压缩块。冷缓存瓶颈常是磁盘或网络带宽，减少读取字节数可能抵消解压 CPU 成本。

该方案会改变 `.b3d` 格式，需要新版本、块长度索引、兼容策略和完整正确性测试，应在批量融合和流水线完成后再评估。

### 15.9 编译和运行时优化

正式提交版建议：

- x64 Release；
- LTO/LTCG；
- 可选 PGO；
- 减少临时分配；
- SIMD 优化块内复制和转置；
- AVX2 使用 CPUID 运行时分派。

官方 CPU 型号不确定，不应让整个程序无条件依赖 AVX2。

## 16. 禁止或不推荐的做法

除非赛事方书面明确允许，不应：

- 在计时前隐藏地完整读取比赛数据；
- 把排除 warm-up 时间的热缓存结果作为正式成绩；
- 依赖常驻后台进程跨测试保留目标数据；
- 使用 RAM disk 冒充磁盘读取；
- 携带管理员级缓存操纵工具作为提交程序的一部分；
- 保存 X/Y/Z 三份完整冗余副本；
- hot 阶段复用 cold 阶段已创建或预分配的输出文件；
- 关闭必要的同步写盘要求来制造虚假性能；
- scrub 失败、读取不完整或文件容量不足时仍把结果标记为 `cold_scrubbed`。

## 17. 可跟随执行的改造 Phase

后续改造按以下 Phase 顺序推进。每个 Phase 都有明确目标、操作项和完成标志；前一个 Phase 验收通过后再进入下一个 Phase。

### Phase 1：建立公共 benchmark 基础结构

**目标**：先建立冷、热测试共用的数据结构和参数契约，不改变核心读取算法。

**操作项**：

1. 新增 benchmark 私有支持模块；
2. 增加 `CacheMode`、`ColdMethod`、`ColdIsolation` 等枚举；
3. 实现 `--cache-mode cold|hot|both`；
4. 保留 `--warm-up` 兼容行为；
5. 建立 `BenchCasePlan` 和稳定的 `plan_hash`；
6. 冷、热阶段复用同一份请求计划；
7. 增加统一 phase 日志格式。

**完成标志**：

- CLI 参数可正确解析和检查冲突；
- 固定种子可生成稳定 plan hash；
- 冷、热阶段打印相同 plan hash；
- 原有 reader 正确性测试继续通过。

### Phase 2：实现 scrub 文件准备工具

**目标**：能够一次性创建并校验可复用的大数据包。

**操作项**：

1. 实现 `cache-prepare` 命令；
2. `--size auto` 默认使用 `1.50 × 物理内存`；
3. 使用确定性伪随机块真实写满文件；
4. 禁止生成稀疏文件或压缩文件；
5. 分块写入，避免占用大量内存；
6. 输出文件大小、耗时和校验信息；
7. 已存在且有效时支持显式复用或覆盖策略。

**完成标志**：

- 本地 32 GB 机器可准备约 48 GB scrub 文件；
- 文件关闭并重新打开后大小正确；
- 创建命令和 benchmark 命令相互独立；
- 创建失败不会留下被误认为有效的文件。

### Phase 3：实现冷缓存冲刷

**目标**：每次 cold benchmark 前可靠完成大数据包冲刷。

**操作项**：

1. scrub 前销毁旧 reader 和大型结果缓冲；
2. 使用分段 mmap 或普通 buffered I/O；
3. 实际逐页触碰完整 scrub 文件；
4. 访问尾页最后一个字节；
5. 计算并输出 checksum；
6. 关闭所有 mapping 和文件句柄；
7. 等待默认 1000 ms；
8. 任一条件失败则终止 cold benchmark。

**完成标志**：

- 输出明确显示 `[Phase 1/5] Cache scrub`；
- bytes、elapsed、checksum 完整可见；
- 成功后状态为 `cold_scrubbed`；
- 失败时不输出 cold 成绩。

### Phase 4：实现双阶段 benchmark

**目标**：一个命令按固定顺序输出冷、热两组结果。

**操作项**：

1. 运行 scrub；
2. 新建 reader 执行 `cold_scrubbed`；
3. 销毁 cold reader；
4. 显式 warm-up；
5. 重新创建 reader 执行 `hot_prefetched`；
6. 输出每轴和汇总结果；
7. 输出 `observed_ratio = cold_time / hot_time`；
8. 支持 `cold`、`hot`、`both` 三种模式。

**完成标志**：

- 冷、热请求顺序完全一致；
- reader 内部缓存起点一致；
- warm-up 时间不计入 hot benchmark；
- 单命令可以完成完整冷/热对比。

### Phase 5：改造 `run_test` 写出路径

**目标**：让含结果写出的 benchmark 也能公平执行冷、热两阶段。

**操作项**：

1. 扩展 `BenchResult`，拆分 `read_sec`、`write_sec`、`total_sec`；
2. 冷、热使用独立输出目录；
3. 两个目录都在计时前创建；
4. 运行前估算输出空间；
5. cold/hot 对应文件进行字节一致性检查；
6. 正式主指标继续使用 `total_sec`；
7. 保留 `_commit`/`fsync` 配置语义。

**完成标志**：

- hot 不覆盖或复用 cold 输出文件；
- 两阶段输出内容一致；
- 空间不足时在 benchmark 前失败；
- 日志清楚区分 read、write、total。

### Phase 6：增强 warm-up 语义

**目标**：让 `warm_up()` 返回时确实完成目标范围触页。

**操作项**：

1. 保留 `PrefetchVirtualMemory`/`madvise` 作为提示；
2. 提示后实际逐页触碰；
3. 访问目标范围最后一个字节；
4. 完成后才设置 warm-up done；
5. 正确报告 `dataset`、`workload`、`partial`、`best_effort`；
6. 增加同步和异步 warm-up 测试。

**完成标志**：

- warm-up 前后读取结果一致；
- 同步和异步状态正确；
- memory cap 不越界；
- hot phase 能明确报告实际预热范围。

### Phase 7：补齐自动测试和文档

**目标**：保证 benchmark 改造不会破坏格式、读取正确性和 CLI 稳定性。

**操作项**：

1. 增加 benchmark 真实进程测试；
2. 验证非法参数和冲突参数；
3. 验证 scrub 文件错误处理；
4. 验证 plan hash 一致；
5. 验证 cold/hot 输出文件一致；
6. 不增加时间快慢断言；
7. 更新根目录和子目录 code map；
8. 记录最终 CLI 使用示例。

**完成标志**：

- `test_block3d`、`test_cli` 和新增 benchmark 测试全部通过；
- Windows 进程错误能正常返回，不发生 fast-fail；
- 文档和实际参数保持一致。

### Phase 8：优化正式冷路径

**目标**：在可信的 `cold_scrubbed` 环境中缩小冷、热性能差距。

**操作顺序**：

1. 批量切片融合；
2. 物理块并集和动态读取策略；
3. 有界读写流水线；
4. 输出缓存和预分配策略；
5. 定向异步预取；
6. 固定线程池；
7. 按不同存储介质选择 block size。

**完成标志**：

- 相同 plan hash 下 cold 总时间稳定下降；
- 实际读取字节数和随机 I/O 数量下降；
- 内存峰值受控；
- X/Y/Z 三轴仍保持均衡；
- 正确性测试和输出校验全部通过。

### Phase 9：高级优化

在前述 Phase 稳定后再评估：

1. 自适应块压缩；
2. SIMD 运行时分派；
3. LTO/PGO；
4. 针对 HDD、SSD、NVMe、集群存储的配置策略。

## 18. 总体验收标准

完成 Phase 1～7 后，应满足：

- 一个命令可对同一数据包输出冷、热两组结果；
- 冷、热请求计划完全一致；
- 本地无需重启即可执行 `cold_scrubbed`；
- scrub 成功后直接按冷缓存口径记录；
- scrub 失败时不继续输出冷缓存成绩；
- warm-up 时间与正式读取时间分开；
- `run_test` 冷、热结果文件不互相覆盖；
- 冷、热结果文件内容完全一致；
- benchmark 不要求管理员权限；
- reader 构造不自动完整预热；
- CTest 不依赖不稳定的性能阈值；
- 原有格式、切片和并发正确性测试继续通过。

进入 Phase 8 后，应重点观察：

- `cold_scrubbed / hot_prefetched` 的时间比值是否缩小；
- 冷缓存下实际读取字节数是否下降；
- 随机 I/O 数量是否下降；
- 连续切片块复用率是否提高；
- 峰值内存是否受控；
- 读取与写出是否有效重叠；
- X/Y/Z 三轴性能是否仍然均衡。

## 19. 最终原则

本地实验环境采用：

```text
大于物理内存的无关文件逐页读取
  -> cold_scrubbed
  -> 显式 workload/full warm-up
  -> hot_prefetched
```

只要 scrub 大数据包按成功判定规则完整读取，该轮就按冷缓存环境使用，不再执行额外校准。

正式赛事优化不依赖隐藏预热，而是围绕以下目标展开：

```text
更少的必要读取字节
+ 更少的随机 I/O
+ 更高的块复用
+ 更大的连续物理范围
+ 读取/提取/写出流水线
+ 受控内存与线程调度
```

热缓存结果用于确定算法上限，`cold_scrubbed` 结果用于指导正式冷路径优化。后续日常实验只需要稳定执行这两种口径。
