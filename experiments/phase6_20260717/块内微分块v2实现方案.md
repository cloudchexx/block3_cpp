# 块内微分块 v2 实现方案

> 目标：在保持宏块 Morton/Z-order 物理顺序和一宏块一 offset 表结构基本不变的前提下，新增一种块内 micro-tile 布局，用于降低 Y/Z 切片的块内访问放大，重点改善 `test50` random Y/Z 性能。
>
> 本文是实现前的设计与执行清单。后续实现、测试和性能 A/B 均以本文为准。

---

## 1. 背景与判断

当前 `.b3d` v1 布局为：

```text
FileHeader 64B
uint64_t block_offsets[total_blocks]   # 逻辑 X-Y-Z 块顺序
padding to 4096-byte boundary
fixed-size macro blocks                 # 宏块按 Morton/Z-order 物理顺序写入
```

宏块内部为局部 X-Y-Z 顺序，Z 最内层连续：

```text
local_offset = lx * block_size * block_size + ly * block_size + lz
```

这种布局对 X 切片和 Z 列比较友好，但对 Y/Z 切片不友好：

- X slice：固定 `lx`，读取一个局部 Y-Z 平面，块内相对连续；
- Y slice：固定 `ly`，每个 `lx` 读取一段连续 Z，`lx` 间跨大 stride；
- Z slice：固定 `lz`，在块内跨 stride 标量采样，最不连续。

最近日志显示 `test50` 的 random Y/Z 明显慢于 X，并且 hot cache 对 Y/Z random 没有稳定收益，说明瓶颈更像是块内访问放大、页/cache line 利用率差、抽取/重组成本，而不是单纯磁盘冷缓存 miss。

因此，块内微分块是下一阶段值得验证的布局优化。

---

## 2. 总体策略

推荐第一版做 **固定可计算 micro-tile 布局**：

```text
宏块尺寸不变
宏块 offset 表不变：仍然一宏块一绝对文件 offset
宏块物理顺序不变：仍然按 Morton/Z-order 排列宏块
宏块内部改为固定 micro-tile 顺序
不增加 per-micro-block offset 表
不引入压缩
不引入多副本
```

第一版建议：

```text
micro_size = 8
```

原因：

- `test18` 当前自动块大小常见为 `56 = 7 * 8`；
- `test50` 当前自动块大小常见为 `64 = 8 * 8`；
- `8^3` micro-tile 为 512 floats，即 2048 bytes；
- micro-tile 粒度小于 4KiB page，但不会像 `4^3` 那样导致过多循环和分支；
- `16^3` 又偏粗，对 Z slice 的改善可能不足。

## 3. 明确目标与非目标

### 3.1 目标

1. 新增 `.b3d` v2 micro-tiled 块内布局。
2. v1 文件继续可读，保持历史兼容。
3. v2 文件由转换器显式生成，reader 按 header/layout 正确识别。
4. 重点改善：
   - `test50` random Y slice；
   - `test50` random Z slice；
   - `test50` random Avg / Total elapsed。
5. 保证正确性：
   - 三轴 slice；
   - batch fused slice；
   - column；
   - subvolume；
   - point；
   - verify。
6. 存储 ratio 基本不变，不能因为新增索引明显膨胀。

### 3.2 非目标

第一版不做：

- per-micro-block offset 表；
- 压缩/解压；
- X/Y/Z 多副本布局；
- GPU；
- 动态选择不同 micro_size；
- 对每个 axis 分别设计不同块内布局；
- 改变宏块 Morton/Z-order 物理顺序；
- 改变公共 API 的输出布局。

## 4. 文件格式设计

### 4.1 Header 约束

现有 `FileHeader` 固定 64 字节：

```cpp
struct FileHeader {
    char     magic[4];
    uint32_t version;
    uint64_t dim_x;
    uint64_t dim_y;
    uint64_t dim_z;
    uint64_t block_size;
    uint64_t total_blocks;
    uint64_t data_offset;
    uint64_t reserved;
};
```

实现时保持：

```text
sizeof(FileHeader) == 64
索引表仍紧随 header
数据区仍 4096 对齐
```

### 4.2 版本与 layout 标记

推荐：

```text
version = 1: legacy xyz layout
version = 2: header.reserved 编码 layout 信息
```

`reserved` 建议编码为：

```text
低 8 bit: layout_kind
    0 = legacy xyz
    1 = micro_tiled_xyz

次低 8 bit: micro_size
    0 = 无 micro tile
    8 = 8x8x8 micro tile

其余 bit: 预留，必须为 0
```

伪代码：

```cpp
constexpr uint32_t VERSION_LEGACY = 1;
constexpr uint32_t VERSION_MICROTILE = 2;

constexpr uint64_t LAYOUT_LEGACY_XYZ = 0;
constexpr uint64_t LAYOUT_MICRO_TILED_XYZ = 1;

inline uint64_t encode_layout(uint8_t layout_kind, uint8_t micro_size) {
    return static_cast<uint64_t>(layout_kind)
         | (static_cast<uint64_t>(micro_size) << 8);
}

inline uint8_t header_layout_kind(uint64_t reserved) {
    return static_cast<uint8_t>(reserved & 0xffu);
}

inline uint8_t header_micro_size(uint64_t reserved) {
    return static_cast<uint8_t>((reserved >> 8) & 0xffu);
}
```

兼容规则：

| 文件 | reader 行为 |
|---|---|
| `version == 1` | legacy XYZ 块内布局；允许 `reserved == 0`。 |
| `version == 2 && layout_kind == 1 && micro_size == 8` | micro-tiled v2。 |
| 其他 version/layout | 报错，不猜测读取。 |

## 5. Micro-tile 块内布局

### 5.1 基本定义

设：

```text
B = block_size
M = micro_size，第一版固定 8
micro_count = ceil(B / M)
```

对于局部坐标：

```text
lx, ly, lz in [0, B)
```

拆成 micro 坐标和 micro 内坐标：

```text
mx = lx / M
my = ly / M
mz = lz / M

ix = lx % M
iy = ly % M
iz = lz % M
```

第一版要求 `block_size % micro_size == 0`，否则转换器拒绝生成 v2。这样可以避免边界 micro-tile 在宏块内部再出现不规则尺寸。

由于当前自动块大小 `56`、`64` 均可被 8 整除，这个限制对当前目标数据集可接受。

### 5.2 Micro-tile 顺序

第一版 micro-tile 之间仍采用 X-Y-Z 展平，micro 内部仍 Z 连续：

```text
micro_linear = mx * micro_count * micro_count
             + my * micro_count
             + mz

inner_linear = ix * M * M
             + iy * M
             + iz

micro_tile_floats = M * M * M

local_offset_v2 = micro_linear * micro_tile_floats + inner_linear
```

这种布局不是为了让某一个轴绝对最优，而是降低单个宏块中 Y/Z 平面的跨页/跨 cache line 访问跨度。

### 5.3 可选备选：micro-tile Morton 顺序

若第一版效果不理想，可追加第二个实验布局：

```text
micro_linear = morton_encode(mx, my, mz) 在 micro_count 范围内排序后的 rank
```

但第一版不建议直接使用 micro-level Morton，因为：

- 需要 rank 映射或稳定排序；
- 实现复杂度更高；
- 对 batch/window 的可解释性较差；
- 第一轮目标是验证“块内微分块是否有收益”，不是一次性搜索最优布局。

## 6. 实现改动清单

### 6.1 公共格式与布局层

涉及文件：

- `include/block3d/types.hpp`
- `include/block3d/core.hpp`
- `include/block3d/code_map.md`

需要新增：

1. version/layout 常量：

```cpp
constexpr uint32_t VERSION_LEGACY = 1;
constexpr uint32_t VERSION_MICROTILE = 2;
constexpr uint64_t LAYOUT_LEGACY_XYZ = 0;
constexpr uint64_t LAYOUT_MICRO_TILED_XYZ = 1;
```

2. header layout 编解码 helper。
3. 块内 offset helper：

```cpp
inline uint64_t legacy_local_offset(uint64_t bs,
                                    uint32_t lx, uint32_t ly, uint32_t lz);

inline uint64_t micro_tiled_local_offset(uint64_t bs, uint32_t micro_size,
                                         uint32_t lx, uint32_t ly, uint32_t lz);
```

4. layout 枚举：

```cpp
enum class BlockInnerLayout {
    LegacyXYZ,
    MicroTiledXYZ,
};
```

原则：

- `BlockLayout3D::linear_index()` 不改；
- `block_floats` 和 `block_bytes` 不改；
- v2 只改变宏块内部 offset 解释。

### 6.2 Converter

涉及文件：

- `src/converter.cpp`
- `include/block3d/converter.hpp`
- `src/cli.cpp`

建议新增 converter option：

```cpp
struct ConvertOptions {
    uint64_t block_size = 0;
    uint64_t max_memory_mb = 0;
    BlockInnerLayout inner_layout = BlockInnerLayout::LegacyXYZ;
    uint32_t micro_size = 0;
};
```

如果不想第一轮大改 API，可保留原函数并新增重载：

```cpp
void convert_raw_to_blocked(...);  // 默认 legacy
void convert_raw_to_blocked(..., const ConvertOptions& options);
```

CLI 建议新增参数：

```text
--layout legacy|micro-tiled
--micro-size N
```

规则：

- 默认仍为 `legacy`，避免破坏现有流程；
- `--layout micro-tiled` 默认 `--micro-size 8`；
- `micro_size` 必须为 2 的幂或至少为正数；第一版实际只允许 8；
- `block_size % micro_size == 0`，否则报错；
- micro-tiled 文件写 `version=2`，`reserved=encode_layout(1, micro_size)`。

核心改动在 `extract_block()`：

当前 legacy：

```cpp
dst = lx * bs * bs + ly * bs + lz
```

v2：

```cpp
dst = micro_tiled_local_offset(bs, micro_size, lx, ly, lz)
```

边界块仍然：

- 整个固定大小宏块先补零；
- 只写有效 `nx, ny, nz` 范围；
- 读取时仍用 `block_extent()` 裁剪。

### 6.3 Reader 初始化

涉及文件：

- `src/reader.cpp`
- `include/block3d/reader.hpp`

Reader 构造时需要解析：

```text
version
reserved layout_kind
reserved micro_size
```

新增 reader 成员：

```cpp
BlockInnerLayout inner_layout_ = BlockInnerLayout::LegacyXYZ;
uint32_t micro_size_ = 0;
```

初始化规则：

```text
version 1 -> LegacyXYZ, micro_size=0
version 2 + layout_kind micro_tiled -> MicroTiledXYZ, micro_size=8
其他 -> throw runtime_error
```

新增统一块内寻址函数：

```cpp
uint64_t local_offset(uint32_t lx, uint32_t ly, uint32_t lz) const;
```

或静态 helper：

```cpp
uint64_t local_offset_for_layout(BlockInnerLayout layout,
                                 uint64_t block_size,
                                 uint32_t micro_size,
                                 uint32_t lx, uint32_t ly, uint32_t lz);
```

### 6.4 Slice 读取路径

涉及文件：

- `src/reader.cpp`

必须同步修改：

- `read_x_slice()`
- `read_y_slice()`
- `read_z_slice()`
- `read_slices_batch_stream()`

#### X slice

legacy 当前可以对每个 `ly` 连续 `memcpy(nz)`。

micro-tiled 下，如果 `lz` 范围跨多个 micro-tile，不能再用单次大 memcpy 覆盖整行。建议第一版实现简单正确的分段 copy：

```text
对 ly
  对 lz 按 micro boundary 分段
    memcpy 当前 micro 内连续 z 段
```

#### Y slice

类似 X slice：

```text
对 lx
  对 lz 按 micro boundary 分段
    memcpy 当前 micro 内连续 z 段
```

#### Z slice

legacy 当前是标量取值。micro-tiled 后仍可先用标量逻辑实现，确保正确性；后续可优化为 micro-tile 内小块拷贝：

```text
对 lx
  对 ly
    dst = src[local_offset(lx, ly, lz)]
```

第一版可以接受 Z slice 仍然逐点取，只要地址局部性改善后整体更快。

#### Batch fused

`read_slices_batch_stream()` 内部有三轴填充逻辑，必须使用与单 slice 相同的 local offset helper，避免 v1/v2 行为分叉出错。

建议重构出内部 copy helper：

```cpp
copy_x_slice_from_block(...)
copy_y_slice_from_block(...)
copy_z_slice_from_block(...)
```

单片和 batch 共用 helper，降低维护风险。

### 6.5 Column / Subvolume / Point

涉及文件：

- `src/reader.cpp`

必须修改：

- `read_x_column()`
- `read_y_column()`
- `read_z_column()`
- `read_subvolume()`
- `read_point()`

注意：

- `read_z_column()` legacy 当前能对块内连续 Z 段 `memcpy`；micro-tiled 后需要按 micro Z 边界分段 `memcpy`；
- `read_subvolume()` 也需要按 micro boundary 分段复制连续 Z 段；
- `read_point()` 直接使用统一 `local_offset()`。


`info` 输出应包含：


### 7.5 错误路径测试

- `--layout micro-tiled --micro-size 7` 报错；

- 未知 version/layout 文件报错。

## 8. 性能观测指标

当前日志不足以直接解释 micro-tile 收益，建议在正式 A/B 前补充轻量统计。

### 8.1 Page touch 估算

在 reader 内部按访问地址估算 unique page：

```text
page_id = (block_file_offset + local_offset_bytes) >> 12
```

按 case 输出：

```text
axis=x/y/z
mode=random/sequential
cache=cold/hot
avg_pages_per_slice
max_pages_per_slice
```

目标是确认：

- legacy 下 Y/Z 是否触碰明显更多 page；
- micro-tiled 下 Y/Z 的 page touch 是否下降；
- X 的 page touch 增加是否可接受。

### 8.2 Reader 内部耗时拆分

可选增加诊断字段：

```text
block_list_sec
block_copy_sec
consumer/write_wait_sec
```

用于区分收益是否发生在 reader 抽取侧，而不是被 writer/pipeline 淹没。

## 9. A/B 实验方案

### 9.1 构建两个版本数据

建议输出不同文件，避免覆盖现有基线：

```text
Z:\wutan\block3d-data\test18_legacy.b3d
Z:\wutan\block3d-data\test18_micro8.b3d
Z:\wutan\block3d-data\test50_legacy.b3d
Z:\wutan\block3d-data\test50_micro8.b3d
```

若现有 `test18.b3d` / `test50.b3d` 是 legacy，可直接作为 legacy baseline，但必须确认 version/layout。

### 9.2 正确性先行

每个 v2 文件先跑：

```bash
build/Release/block3d_cli verify Z:/wutan/block3d-data/test18_micro8.b3d Z:/wutan/block3d-data/test18.dat
build/Release/block3d_cli verify Z:/wutan/block3d-data/test50_micro8.b3d Z:/wutan/block3d-data/test50.dat
```

通过后再跑性能。

### 9.3 正式性能口径

主指标使用 `run_test` 的 `read_write_total`，不是 `block3d_cli bench` 的 read-only。

每组至少 3 轮：

```text
dataset: test18, test50
layout: legacy, micro8
cache-mode: both
cold-method: scrub
warmup-scope: workload
batch-read: fused
pipeline: on
pipeline-memory: 256
read-dispatch: round-robin
output-sync: requested
```

比较方式：

- 同一 `(dataset, layout, cache, mode, axis)` 的 `PLAN_HASH` 必须一致；
- 取 median，不看单轮最佳；
- 主看 `Total elapsed`；
- 次看 random/sequential 的三轴 `Avg`；
- pipeline 模式下不要用 `read_sec + write_sec` 代替 `total_sec`。

### 9.4 判定标准

#### test50 成功标准

期望：

```text
random Y/Z 明显下降
random Avg 下降
Total elapsed 中位数下降
X 不出现灾难性退化
```

可以接受：

```text
X random 小幅变慢
Z column 小幅变慢
sequential 小幅波动
```

不能接受：

```text
test50 Total elapsed 不降
test18 Total elapsed 明显退化
X 轴退化超过 Y/Z 收益
存储 ratio 明显上升
```

#### test18 守门标准

```text
test18 Total elapsed 中位数退化不超过 3%~5%
```

如果 `test50` 提升很大但 `test18` 轻微退化，可考虑保留 layout 选择开关，而不是默认切换。

## 10. 推荐实施顺序

### Phase 1：格式与 helper

1. 新增 version/layout 常量。
2. 新增 header layout 编解码。
3. 新增 legacy/micro local offset helper。
4. 增加小单元测试验证 offset 公式。

### Phase 2：converter 生成 v2

1. 新增 convert options 或重载。
2. CLI 增加 `--layout`、`--micro-size`。
3. `extract_block()` 支持 micro-tiled 写入。
4. `info` 输出 version/layout/micro_size。
5. 测试 v2 文件 header 正确。

### Phase 3：reader 支持 v2

1. reader 解析 v1/v2。
2. 新增 `inner_layout_`、`micro_size_`。
3. `read_point()` 先支持 v2。
4. 三轴单 slice 支持 v2。
5. column/subvolume/full/verify 支持 v2。

### Phase 4：batch fused 支持 v2

1. `read_slices_batch_stream()` 支持 v2。
2. 单片和 batch 尽量共用 copy helper。
3. 并发缓存和 dispatch 语义不变。

### Phase 5：测试补齐

1. `test_block3d` 全覆盖。
2. `test_cli` 覆盖新参数和错误路径。
3. 构建 Release。
4. 跑完整 CTest。

推荐命令：

```bash
cmake --build build --config Release --target test_block3d test_cli
ctest --test-dir build -C Release -R '^(test_block3d|test_cli)$' --output-on-failure
```

### Phase 6：大数据 A/B

1. 生成 `test18_micro8.b3d`、`test50_micro8.b3d`。
2. verify。
3. 每组至少 3 轮 `run_test both`。
4. 归档日志到新的实验目录，例如：

```text
experiments/phase6_YYYYMMDD/
```

5. 生成 CSV/TXT 汇总。
6. 根据 median 决定是否默认启用 micro-tiled。

## 11. 回滚策略（做好jj版本管理）

必须保持：

- 默认 `convert` 仍生成 legacy v1，除非 A/B 后明确切换；
- reader 始终支持 v1；
- v2 通过 CLI 参数显式启用；
- `run_test` 默认数据文件路径不自动替换为 v2；
- 如性能不达标，可仅保留 v2 实验代码或完全回滚，不影响现有成绩。

若决定默认切换到 v2，需要单独更新：

- 根目录 `CLAUDE.md` 的 `.b3d` 格式说明；
- `code_map.md`；
- `include/block3d/code_map.md`；
- `src/code_map.md`；
- 基准操作手册；
- 实验归档 README。

## 12. 风险清单

| 风险 | 说明 | 缓解 |
|---|---|---|
| X slice 退化 | legacy 对 X 平面很友好，micro-tile 会拆碎连续平面 | A/B 中 X 轴单独设阈值；保留 legacy 默认和 CLI 开关 |
| Z column 退化 | legacy 下 Z column 可大段 memcpy | micro-tiled 下按 micro Z 分段 memcpy；测试 column correctness/perf |
| Batch fused 复杂化 | `read_slices_batch_stream()` 有独立填充逻辑 | 抽公共 block copy helper，单片和 batch 共用 |
| 格式兼容错误 | v1/v2 混读可能静默错 | version/layout 严格校验；未知格式直接报错 |
| 边界块错误 | 宏块边界补零 + micro offset 容易错 | 非整维度测试覆盖所有 API |
| 存储膨胀 | 新索引或 padding 可能影响 storage score | 第一版不增加 micro offset 表，宏块大小不变 |
| 性能收益被 writer 吞掉 | 官方口径包含写文件 | 同时看 reader 内部诊断和 `run_test total_sec` |
| page 粒度收益不足 | mmap cold read 以 OS page 为单位 | 加 page touch 估算验证理论收益 |

---

## 13. 最终决策点

完成 Phase 6 后，根据以下结果决定下一步：

### 13.1 默认启用 micro-tiled

条件：

```text
test50 Total elapsed median 明显下降
test50 random Y/Z 明显下降
test18 Total elapsed median 退化 <= 3%~5%
正确性测试全通过
存储 ratio 基本不变
```

### 13.2 保留为可选实验布局

条件：

```text
test50 有局部收益，但整体不稳定
或 test18 有一定退化
或 X 轴退化较明显
```

### 13.3 回滚/放弃

条件：

```text
test50 Total elapsed 不降
或 correctness/format 复杂度过高
或 X/test18 退化不可接受
```

## 14. 实现备注

- 第一版优先正确性和可比性，不要过早做复杂 SIMD/vectorization。
- 不要在实现 micro-tile 的同时修改 dispatch 默认策略，避免 A/B 因素混杂。
- 不要同时修改 auto block size 策略，先固定验证 layout 影响。
- 不要用 `block3d_cli bench` 的 read-only 结果替代 `run_test` 的正式 read-write total 结果。
- 新增性能日志字段时，应保持旧字段稳定，避免破坏已有解析脚本。
- 所有报告中的性能结论必须说明 cold/hot、scrub、warmup、PLAN_HASH 和 median 口径。
