# Block3D 批量融合读取与读写流水线优化开发方案

> 状态：已确认范围，作为后续优化开发的指导文档  
> 基线日期：2026-07-16  
> 适用项目：`block3d-cpp`  
> 主要目标：在不改变 `.b3d` 格式、自适应块大小和正式写出语义的前提下，降低三轴切片读取与写出总时间，并控制批量读取的峰值内存。

---

## 1. 已确认的技术决策

本方案以下列决定为前提，后续 Phase 不再重复讨论这些方向。

| 事项 | 决定 |
|---|---|
| 正式写出语义 | 保持现状：每张切片依次执行完整 `fwrite`、`fflush`、`_commit/fsync`、`fclose`；任一步失败都不计为成功写出 |
| `read_slices_batch()` | 改造成真正按 block layer 融合的批量读取，而不是逐张调用 `read_slice()` |
| 读取与写出 | 改造成有界 producer-consumer 流水线，读取下一批时并行写出上一批 |
| Reader 块处理线程 | 使用 reader 生命周期内的持久线程池，不再每张切片反复创建和销毁线程 |
| 块分发策略 | 开发阶段对比 round-robin 与 contiguous physical chunks，依据正式基准结果确定最终默认策略 |
| 自适应 block size | 不修改 `auto_block_size()`，不要求评审现场进行参数搜索或运行时调优 |
| 磁盘格式 | 不修改 `FileHeader`、索引表、Morton 宏块布局和块内 X-Y-Z 布局 |
| 块内微分块 | 本轮不做 |
| 隐藏预热 | 禁止；仅显式 hot phase 可以预热 |
| GPU、压缩和多副本 | 不在本轮范围内 |

### 1.1 不可破坏的不变量

1. `.b3d` 仍是 64 字节头、逻辑块偏移表、4096 字节对齐的数据区和 Morton 物理块顺序。
2. 块内仍为局部 X-Y-Z 顺序，Z 连续。
3. 切片输出布局保持：X 为 Y-Z，Y 为 X-Z，Z 为 X-Y。
4. `read_slices_batch()` 返回结果的顺序必须与输入 `indices` 一致，包括重复和乱序请求。
5. `sorted_block_list()` 必须继续按值返回；不能在释放 `cache_mutex_` 后持有缓存内部 `vector` 引用。
6. cold/hot 必须复用同一请求计划并保持相同 `PLAN_HASH`。
7. `run_test` 主指标仍是包含读取、创建输出文件、写入、同步和关闭的 `total_sec`。
8. `output_sync=requested` 时，每个输出文件仍必须独立完成持久化请求，不能改成 phase 末尾统一同步。
9. 不修改 `src/core.cpp::auto_block_size()`，也不在程序启动时自动进行 block size 或 dispatch 性能搜索。
10. 不依赖执行前隐藏预热、预先读取数据文件或保留上一次运行缓存。

当前约束的主要代码位置：

- `include/block3d/reader.hpp`：公共 reader API 和块列表缓存；
- `src/reader.cpp::for_each_block_parallel()`：当前每次调用创建线程的块并行路径；
- `src/reader.cpp::sorted_block_list()`：同一 `(axis, index / block_size)` 的物理排序块列表；
- `src/reader.cpp::read_slices_batch()`：当前逐张顺序派发路径；
- `src/run_test.cpp::run_bench()`：当前“全部读完，再全部写出”路径；
- `src/run_test.cpp::sync_output()`：正式逐文件同步语义。

### 2.1 当前实现的主要限制

1. `read_slices_batch()` 逐张调用 `read_slice()`，同一 block layer 会被重复遍历。
2. 单张切片调用内部会创建并 join 一组 `std::thread`，高频批量读取存在重复线程生命周期成本。
3. `run_bench()` 先保存整批结果，再开始写盘，读取与写出完全串行。
4. `test18` X 单片大小为：

```text
2405 × 2501 × sizeof(float)
= 24,059,620 bytes
≈ 22.94 MiB
```

100 张随机 X 切片仅结果数据就约 2.24 GiB，超出合理的流水线缓冲规模。
5. 已按物理偏移排序的块当前采用 round-robin 分给线程，每个线程自身的访问序列并不连续；是否优于连续物理区间分片尚未通过正式 A/B 证明。

## 3. 总体架构

目标数据流：

```text
固定请求计划 indices
  -> 按 (axis, index / block_size) 分组
  -> 同一 block layer 内按有界 window 融合读取
  -> 完整 SliceJob 移交给有界队列
  -> 单个专用 writer 线程逐文件 fwrite
  -> 每文件 fflush + _commit/fsync + fclose
  -> 统计 total/read/write/queue timing
```

线程关系：

```text
Benchmark 主线程 / producer
  └─ BlockedFileReader 持久线程池
       └─ 并行处理一个 layer window 的物理块

Writer 线程
  └─ 独占每个 SliceJob
       └─ open -> write -> sync -> close
```

### 3.1 Phase 依赖关系

```text
Phase 0  基线、指标和正确性护栏
   ↓
Phase 1  block-layer 融合批量读取
   ↓
Phase 2  流式 API 与有界读写流水线
   ↓
Phase 3  持久线程池（先保持 round-robin）
   ↓
Phase 4  contiguous 与 round-robin A/B，选择默认策略
   ↓
Phase 5  正式回归、性能验收和文档收敛
```

此顺序有意先验证融合读取语义，再改变线程执行器。每个 Phase 只引入一类主要变量，便于定位正确性或性能回归。

## 5. Phase 1：真正的 block-layer 融合批量读取

### 目标

将 `read_slices_batch()` 从逐张派发改为：同一 `(axis, index / block_size)` 的多个切片共享一次块列表获取和一次或少数几次物理块遍历。

### 涉及文件

- `include/block3d/reader.hpp`
- `src/reader.cpp`
- `tests/test_block3d.cpp`
- `include/block3d/code_map.md`
- `src/code_map.md`

### 5.1 请求模型

内部请求描述：

```cpp
struct SliceRequest {
    size_t request_pos;  // indices 中的原始位置
    uint64_t index;      // 全局切片坐标
    uint32_t layer;      // index / block_size_
    uint32_t local;      // index % block_size_
};
```

处理步骤：

1. 在开始产生任何结果前校验 axis 和全部 index，避免半批成功。
2. 按 `layer` 分组，同时保留每项的 `request_pos`。
3. layer 组按首次出现顺序处理，保证运行确定性。
4. 每个 layer 只调用一次 `sorted_block_list(axis, layer * block_size_)`。
5. 每个 layer 再按 `window_slices` 切成有界窗口。
6. 对一个窗口分配若干完整输出切片。
7. 遍历该 layer 的物理排序块；每访问一个块，同时填充窗口内所有目标 local slice。
8. 兼容 API 将结果放回 `results[request_pos]`，保证最终返回顺序与输入一致。

### 5.2 三轴融合内核

#### X slice

- 固定 `bx=layer`；
- 一个窗口包含多个 `lx`；
- 一个 `(bx, by, bz)` 块被访问一次；
- 对每个 `lx` 把块内 Y-Z 平面复制到相应输出切片。

#### Y slice

- 固定 `by=layer`；
- 一个窗口包含多个 `ly`；
- 一个 `(bx, by, bz)` 块被访问一次；
- 对每个 `ly` 和局部 X 复制连续 Z 段。

#### Z slice

- 固定 `bz=layer`；
- 一个窗口包含多个 `lz`；
- 一个 `(bx, by, bz)` 块被访问一次；
- 内层循环应以同一次源缓存行访问服务多个 `lz`，但不改变块内格式。

### 5.3 重复、乱序和完成顺序

- 兼容版 `read_slices_batch()` 必须返回与请求完全相同的顺序。
- 重复 index 必须生成独立结果，不能向调用者暴露共享可变 buffer。
- 后续流式 API 可以按 layer/window 完成顺序回调，不保证全局请求顺序；必须携带 `request_pos`，由使用者生成正确文件名和统计槽位。
- 文件最终内容和命名必须正确，物理写出先后顺序不是公共 API 契约。

### 5.4 公共 API 与共享内部实现

保留现有兼容 API：

```cpp
std::vector<std::vector<float>>
read_slices_batch(char axis,
                  const std::vector<uint64_t>& indices,
                  int num_threads = 0);
```

新增流式入口，供 Phase 2 使用：

```cpp
struct SliceBatchOptions {
    int num_threads = 0;
    size_t window_slices = 1;
};

using SliceConsumer = std::function<void(
    size_t request_pos,
    uint64_t index,
    std::vector<float>&& slice)>;

void read_slices_batch_stream(char axis,
                              const std::vector<uint64_t>& indices,
                              const SliceBatchOptions& options,
                              const SliceConsumer& consumer);
```

实现要求：

- 两个公共入口必须复用同一个 fused engine；
- 兼容 API 只是把 stream 结果收集到 `results[request_pos]`；
- `run_test` 在 Phase 2 后不得调用会收集全部结果的兼容 API；
- `num_threads=0` 表示使用 reader 的线程配置；非零值在 Phase 3 后作为该调用的 worker 上限，而不是临时创建新线程。

### 5.5 并发和异常规则

1. 块列表仍按值返回，融合任务在 `cache_mutex_` 外执行。
2. 同一窗口中，不同物理块写入各输出切片的不重叠区域，无需对输出数组加锁。
3. worker 首个异常被保存；当前窗口全部任务停止或安全汇合后，在调用线程重新抛出。
4. consumer 抛异常时停止后续窗口；已经 move 给 consumer 的切片由 consumer 所有。
5. 不允许返回部分填充的切片作为成功结果。

### 5.6 正确性验收

必须新增或强化：

- 三轴同 layer 融合；
- 三轴跨 layer；
- 重复与乱序；
- 边界块和最后一个合法 index；
- `window_slices=1` 与普通 `read_slice()` 一致；
- `window_slices>1` 与 `window_slices=1` 逐元素一致；
- 同一个 reader 的多线程、混合轴调用；
- `test_concurrent_adjacent_same_block_key()` 全部通过。

### 5.7 性能验收

以 read-only 为主，避免输出写盘掩盖融合收益：

- 同一 layer 的 8 张连续切片，三轴 read-only 中位时间目标至少改善 15%；
- 100 张随机切片的任何轴不得回退超过 5%；
- 单张切片路径不得回退超过 3%；
- 如果某轴未改善，必须记录其 layer 分布、实际窗口数和块遍历次数，不能只看总时间猜测。

### 回滚点

- `read_slice()` 单片路径保持不变；
- 开发期可通过 `--batch-read legacy` 回退逐张派发；
- 融合内核、API 和测试使用独立提交，发现问题时不连带回滚后续无关改动。

## 6. Phase 2：流式 API 与有界读写流水线

### 目标

消除 `run_test` 的“整批读取完成后才开始写出”，使读取下一窗口与写出上一窗口重叠，同时严格限制完整切片 buffer 的数量。

### 涉及文件

- `src/run_test.cpp`
- `include/block3d/reader.hpp`
- `src/reader.cpp`
- `tests/test_cli.cpp`（若公共 CLI 参数发生变化）
- `src/code_map.md`
- `基准操作手册.md`（完成并验证后更新）

### 6.1 Pipeline 数据结构

建议使用以下内部概念，实际命名应匹配周边代码风格：

```cpp
struct SliceJob {
    size_t request_pos;
    uint64_t index;
    fs::path output_path;
    std::vector<float> data;
};

class BoundedSliceQueue {
public:
    void push(SliceJob&& job); // 队列满时阻塞，形成反压
    bool pop(SliceJob& job);   // 关闭且清空后返回 false
    void close();
    void cancel();
};
```

所有容量必须按**实际切片字节数**推导，而不是只按 vector 个数粗略限制。

### 6.2 内存预算

新增 `run_test` 开发/配置参数：

```text
--pipeline-memory MB
--pipeline-window N
```

推荐初始默认：

```text
pipeline_memory_mb = 256
pipeline_window = auto
writer_threads = 1（本轮固定，不开放多 writer 竞速）
```

推导方式：

```text
slice_bytes = slice_bytes_for_axis(dataset, axis)
max_live_slices = max(1, floor(pipeline_memory_bytes / slice_bytes))
```

需要同时约束 producer 正在构建的 window 和队列中待写的 job：

```text
read_window_slices + queue_capacity_slices <= max_live_slices
```

推荐自动策略：

```text
read_window_slices = min(4, max_live_slices)
queue_capacity_slices = max(1, max_live_slices - read_window_slices)
```

当 `max_live_slices=1` 时，退化为读一张、写一张，不允许突破预算。此时可以没有读写重叠，但仍保证正确性和有界内存。

`test18` X 单片约 24,059,620 bytes。256 MiB 预算最多约 11 张；建议窗口 4 张、队列最多 7 张。任何时刻都不能重新累积 100 张 X 切片。

注意：该预算只约束 pipeline 的完整切片 payload，不应错误描述为进程级硬内存上限。线程栈、块列表、索引表和文件映射虚拟地址空间另计。

### 6.3 `run_bench()` 新流程

Reader、持久线程池和 writer 线程的创建位于计时区间外；正式计时流程为：

```text
t0
  read_slices_batch_stream(...)
    -> 每个 window 进行 fused read
    -> consumer 把完整 SliceJob move 到有界队列
    -> 队列满时 producer 阻塞

  writer 与 producer 同时运行：
    -> fopen
    -> fwrite 全部字节
    -> output_sync 时 fflush + _commit/fsync
    -> fclose

  producer 完成后 close queue
  等待 writer drain 并 join
t1
```

必须继续遵守：reader 构造和 mmap/header parse 不属于 `run_test` timed region，和当前口径一致。

### 6.4 正式写出语义

把现有逐文件写出抽为单一 helper，例如：

```cpp
WriteResult write_slice_file(const SliceJob& job, bool output_sync);
```

helper 必须保持以下顺序：

```text
fopen(path, "wb")
fwrite(data.data(), 1, bytes, file) == bytes
if output_sync:
    fflush(file)
    _commit(_fileno(file)) / fsync(fileno(file))
fclose(file) == 0
```

仅全部成功时才更新：

- `n_written`；
- `total_bytes`；
- `slice_bytes[request_pos]`。

明确禁止：

- 合并多个切片为单个输出文件；
- phase 末尾统一同步；
- 用异步写入后不等待完成；
- 只 `fflush` 而省略 `_commit/fsync`；
- writer 提前报告成功；
- 多线程共同操作同一个 `FILE*`。

默认只使用 **1 个 writer 线程**。多个并发 `_commit/fsync` 可能引起队列竞争和抖动，不在本轮优化范围内。

### 6.5 Pipeline 时序指标

流水线后读取和写出会重叠，旧的：

```text
read_sec + write_sec ≈ total_sec
```

不再成立。因此不能沿用会误导分析的定义。建议输出：

```text
BENCHMARK_RESULT ... total_sec=...
PIPELINE_RESULT producer_wall_sec=...
                writer_service_sec=...
                drain_sec=...
                producer_wait_sec=...
                queue_peak_slices=...
                queue_peak_bytes=...
```

定义：

- `total_sec`：从 producer 开始到 writer 完成最后一个文件同步并关闭；赛事主指标；
- `producer_wall_sec`：producer 完成所有融合读取并提交 job 的墙钟时间，包含因队列满而等待；
- `writer_service_sec`：writer 在各文件 open/write/sync/close 上耗时的累计；
- `producer_wait_sec`：producer 因队列容量反压而等待的累计；
- `drain_sec`：producer 提交完毕后，writer 清空剩余队列所需时间。

为保持历史日志可解析，可暂时继续输出 `read_sec` 和 `write_sec`，但必须在日志中标记：

```text
TIMING_MODEL value=overlapped_pipeline
```

并明确它们是诊断值，不能相加解释 `total_sec`。最终文档和结果分析应以 `total_sec` 为准。

### 6.6 错误传播与取消

1. producer 失败：记录首个异常，关闭/取消队列，等待 writer 安全退出，case 失败。
2. writer 的 open/write/sync/close 任一步失败：记录首个失败路径和错误，设置共享失败标志。
3. producer 在每个 window 和每次入队前检查失败标志，停止产生新结果。
4. 已经由 writer 独占的 job 允许完成清理；不得遗留运行线程或未关闭文件。
5. 任何 case 只要 `n_read != requested` 或 `n_written != requested`，就不输出成功吞吐指标。
6. 所有线程退出后再抛出/报告错误，避免异常展开时 reader、queue 或 mmap 先销毁。

### 6.7 正确性验收

- pipeline on/off 的所有输出逐文件字节一致；
- `output_sync=requested` 与 `disabled` 两条路径均有 smoke 覆盖；
- 文件名、`request_pos`、slice index 和内容正确，即使 fused engine 的完成顺序不同；
- `cache-mode both` 仍得到 `OUTPUT_VERIFY_RESULT ok=1`；
- 人工注入输出目录不可写、写满/短写等失败时，不输出成功成绩且无死锁；
- 空请求、单请求、队列容量 1、窗口 1 和跨 layer 请求均能退出；
- `PIPELINE_RESULT queue_peak_bytes` 不超过配置预算（允许仅有清晰记录的固定元数据误差，不允许 payload 超额）。

### 6.8 性能验收

- test18 random X 100：峰值完整切片 payload 从约 2.24 GiB 降到不超过默认 256 MiB；

## 7. Phase 3：Reader 生命周期内的持久线程池

### 目标

替换 `src/reader.cpp::for_each_block_parallel()` 每次创建和 join `std::thread` 的行为，同时先保持现有 round-robin 分发语义，避免线程池和 dispatch 策略同时变化。

### 涉及文件

- `include/block3d/reader.hpp`
- `src/reader.cpp`
- `tests/test_block3d.cpp`
- 必要时新增私有 `src/thread_pool.hpp/.cpp`
- 若新增 `.cpp`，同步修改 `CMakeLists.txt`

### 7.1 生命周期设计

推荐使用 `BlockedFileReader` 私有 PIMPL：

```cpp
class BlockedFileReader {
public:
    ~BlockedFileReader();

private:
    class ThreadPool;
    std::unique_ptr<ThreadPool> thread_pool_;
};
```

要求：

1. reader 构造时根据 `num_threads_` 创建固定 worker。
2. reader 析构前等待当前任务完成并安全停止所有 worker。
3. 每个公开 read 方法返回前，其提交的块任务必须全部完成。
4. `MappedFile`、块索引和输出 buffer 的生命周期必须覆盖任务执行期。
5. reader 仍不可复制；如果移动语义未定义，不要为线程池强行增加不安全移动。

### 7.2 任务模型

提供同步 barrier 风格的内部接口：

```cpp
parallel_for_blocks(blocks, strategy, process);
```

调用者视角仍是同步函数。实现可以采用 generation counter、共享 work descriptor 和 condition variable，避免为每个块创建 `std::function` 队列对象。

第一阶段保持当前阈值：

```text
num_threads <= 1 或 block_count <= num_threads × 4 -> 串行
```

该阈值后续只能通过独立 A/B 调整，不与线程池首版一起改变。

### 7.3 嵌套和并发调用规则

现有测试会从多个外部线程共享一个 reader 并调用 `read_slice()`。线程池必须明确支持这一点。

推荐规则：

- pool 允许多个调用者排队提交同步 job；
- 单个 job 内的块任务并行；
- 若从 pool worker 内再次调用 pool，必须检测并退化为串行，避免嵌套等待死锁；
- 不在 `cache_mutex_` 下等待 pool；
- 不允许一个调用者覆盖另一个尚未完成的共享 work descriptor。

如果首版线程池只支持一个 active job，则必须用独立调度 mutex 串行化 job 提交，同时确保外部并发调用不会死锁或产生错误结果。性能优化不能以破坏公共 reader 线程安全为代价。

### 7.4 异常规则

- worker 捕获第一个 `std::exception_ptr`；
- 设置当前 job 的取消标志，未开始的块可以跳过；
- active worker 全部抵达 barrier 后，由提交线程重新抛出；
- 下一 job 开始前清除上一 job 的异常和取消状态；
- reader 析构不得抛异常。

### 7.5 正确性验收

- Phase 1/2 全部测试继续通过；
- `test_concurrent_adjacent_same_block_key()` 重复运行无崩溃和死锁；
- 混合 X/Y/Z、单片和 fused batch 的共享 reader 并发测试通过；
- 1、2、4、16 worker 结果逐元素一致；
- worker 中注入异常时能在调用线程观察，随后 reader 仍能执行下一次合法读取，或明确进入不可复用失败态。

### 7.6 性能验收

- hot read-only 单张切片中位数不得回退超过 3%；
- 同一请求计划的 fused batch read-only 目标改善至少 3%，否则至少证明线程创建次数已经从“每张/每窗一次”降为“每个 reader 一次”且无整体回退；
- `run_test total_sec` 三轴总体不得回退超过 3%；
- 记录 pool 创建次数、jobs 和串行 fallback 数量，方便确认代码实际走了预期路径。

### 回滚点

- 保留旧 per-call helper 的开发编译路径或内部策略；
- 首版只替换执行器，不改变分发顺序；
- 线程池实现使用独立提交，便于单独回滚。

## 8. Phase 4：contiguous physical chunks 与 round-robin A/B

### 目标

在持久线程池稳定后，只改变物理排序块列表的 worker 分片方式，用正式 cold/hot 结果选择最终默认策略。

### 8.1 两种策略

当前 round-robin：

```cpp
for (size_t i = worker_id; i < block_count; i += worker_count)
    process(blocks[i]);
```

候选 contiguous：

```text
chunk = ceil(block_count / worker_count)
worker t -> [t * chunk, min((t + 1) * chunk, block_count))
```

`blocks` 已按 `.b3d` 物理偏移排序，因此 contiguous 使每个 worker 自身按一段连续物理区间访问；但线程间并发缺页、负载均衡和 OS 预读效果可能随硬件而不同，不能仅凭理论切换默认值。

### 8.2 参数与默认行为

开发期支持：

```text
--read-dispatch round-robin
--read-dispatch contiguous
```

日志必须输出：

```text
READ_DISPATCH strategy=round_robin
```

评审现场不进行自动 A/B，不根据短暂探测动态选择。Phase 4 结束时，代码中固定一个经过 test18、test50 和目标存储环境验证的默认值；命令行覆盖仅用于诊断和回滚。

### 8.3 A/B 控制变量

每次对比必须固定：

- 相同 `.b3d` 和 `block_size`；
- 相同 binary/build；
- 相同 `PLAN_HASH`；
- 相同线程数；
- 相同 fused window 和 pipeline memory；
- 相同 writer 数（1）；
- 相同 `output_sync=requested`；
- 相同 cold/hot 配置；
- 相同轴和 case 顺序。

正式比较不更改 `auto_block_size()`，也不重新转换出不同 block size 的文件。

### 8.4 试验层级

1. **hot read-only**：观察 CPU 调度、缓存和负载均衡；
2. **cold read-only**：观察 mmap 缺页和物理访问顺序；
3. **正式 read-write total**：观察与单 writer pipeline 共同作用后的最终成绩；
4. test18 通过后在 test50 复验，避免仅针对一个维度过拟合。

### 8.5 选择门槛

切换默认策略必须同时满足：

- test18 与 test50 的三轴等权总体中位数改善至少 3%，或者一个数据集明显改善、另一个基本持平；
- 任一正式 axis/mode 不得稳定回退超过 5%；
- cold 正式成绩不能只靠 hot 改善掩盖；
- 全部输出逐文件一致；
- 至少 3 轮正式运行，使用中位数，不使用最佳单轮；
- 若结果在噪声范围内，保持较简单或当前策略，不做无证据切换。

### 回滚点

`--read-dispatch` 始终可以显式回到另一策略。默认值切换必须是独立提交，不与线程池或融合算法混合。

## 11. 风险清单

| 风险 | 影响 | 约束或缓解 |
|---|---|---|
| 融合窗口太小 | 同一 layer 被多次遍历，收益受限 | 在 payload 预算内自动选择 2～4 张起步，记录窗口数和遍历次数 |
| 融合窗口太大 | 峰值内存反弹 | `window + queue <= max_live_slices`，按字节预算强制反压 |
| Pipeline 指标歧义 | 错把重叠后的 read/write 累加 | 保留 `total_sec` 主指标，新增 `TIMING_MODEL=overlapped_pipeline` 和诊断字段 |
| 单盘读写相互竞争 | pipeline 可能使个别 cold case 回退 | 有界小队列、单 writer，并以正式 total 中位数决策 |
| 多 writer 并发同步抖动 | `_commit/fsync` 竞争、尾延迟增大 | 本轮固定一个 writer 线程 |
| 线程池嵌套死锁 | reader 共享并发调用卡死 | pool worker 内嵌套提交退化串行；多调用者 job 正确排队 |
| 缓存引用失效 | use-after-free 或崩溃 | `sorted_block_list()` 继续按值返回，保留现有回归测试 |
| 完成顺序改变 | 写错文件名或统计槽位 | 每个 job 携带 `request_pos` 和 `index`，按 request_pos 更新结果 |
| writer 错误后 producer 阻塞 | 死锁或内存泄漏 | 共享取消状态、queue cancel、所有线程统一 join |
| 只优化 test18 | test50 或正式维度回退 | Phase 5 必须在 test50 复验 |
| 动态现场调优不稳定 | 评审时结果不可预测 | 不改 auto block size，不做运行时策略竞赛；预先固定默认 |
| 隐式缓存污染 | cold 成绩失真 | 不在 reader/pipeline 内增加 warm-up；验证后重新 scrub |

## 14. 完成定义

本轮优化只有在同时满足以下条件时才算完成：

1. `read_slices_batch()` 已按 block layer 融合，不再逐张重复派发；
2. `run_test` 已使用流式 API 和有界队列，不再一次保存 100 张大切片；
3. 正式写出仍逐文件完整执行 `fwrite + fflush + _commit/fsync + fclose`；
4. reader 使用经过并发测试的持久线程池；
5. round-robin 与 contiguous 已通过受控 A/B，默认策略已在开发阶段固定；
6. 自适应 block size、磁盘格式和块内布局未改变；
7. test18/test50 正确性通过，cold/hot 输出一致；
8. 正式三轴等权成绩无回退，并尽可能取得可复现改善；
9. 峰值切片 payload 受配置预算约束；
10. 评审现场无需额外调参、预热或性能搜索即可直接运行默认路径。
